# 餐點條件篩選抽卡

**建議功能分支**: `005-card-metadata-filtered-draw`  
**建立日期**: 2026-05-14  
**狀態**: 草稿  
**輸入**: 使用者已選擇「產品實用性優先」，本版先新增餐點卡牌決策資訊與條件篩選抽卡，讓抽卡結果更符合當下吃飯限制與偏好。

## Clarifications

### Session 2026-05-14

- Q: 下一個「產品實用性優先」功能要鎖定哪個範圍？ → A: 條件篩選抽卡，新增卡牌 metadata 與首頁/卡牌庫篩選。

## 使用者情境與測試 *(mandatory)*

### 使用者故事 1 - 依當下條件篩選後公平抽卡 (Priority: P1)

當使用者不知道要吃什麼，但當下有明確限制，例如預算、時間、想吃蔬食、不想吃辣或只想吃特定類型餐點時，可以在首頁設定條件後再抽卡。系統必須只從符合條件的有效卡牌中抽出結果，並在候選池內維持等機率，讓使用者既能縮小選擇範圍，也不破壞既有公平抽卡規則。

**優先級理由**: 這是「產品實用性優先」的核心價值。既有抽卡已能公平隨機，但缺少決策條件會讓結果可能不符合當下需求；條件篩選能直接提高抽卡結果可採用率。

**獨立測試**: 可準備多張具備不同價格、時間、飲食偏好、辣度與標籤的卡牌，在首頁套用條件後分別執行正常模式與隨機模式抽卡；只要候選池只包含符合條件的 active cards，且候選池內每張卡牌標稱機率仍為 `1/N`，此故事即可獨立交付價值。

**驗收情境**:

1. **Given** 使用者選擇正常模式、午餐、低價位、快速、非辣條件，**When** 使用者完成投幣並啟動抽卡，**Then** 系統只會從同時符合午餐與所有篩選條件的有效卡牌中等機率抽出一張。
2. **Given** 使用者選擇隨機模式並套用「蔬食」與「低辣度」條件，**When** 使用者啟動抽卡，**Then** 系統不要求餐別，並從早餐、午餐、晚餐全部符合條件的有效卡牌中等機率抽出一張。
3. **Given** 使用者套用的條件沒有任何符合的有效卡牌，**When** 使用者嘗試抽卡，**Then** 系統不得產生抽卡結果或成功歷史，並清楚提示目前沒有符合條件的餐點。
4. **Given** 使用者切換語系、主題或 reduced motion 設定，**When** 使用者以相同條件抽卡或重顯結果，**Then** 系統不得改變候選池、抽中卡牌身分、歷史紀錄或統計口徑。

---

### 使用者故事 2 - 以決策條件瀏覽與搜尋卡牌 (Priority: P2)

當使用者想主動比較餐點而不是立刻抽卡時，可以在卡牌庫用關鍵字、餐別與決策條件組合篩選，例如找出晚餐、低價位、可外帶、不辣的餐點。系統要讓使用者看懂目前套用哪些條件，並能快速清除或調整條件。

**優先級理由**: 卡牌庫是抽卡以外的主要決策入口；若 metadata 只能用於抽卡，使用者仍難以管理與理解自己的卡牌內容。

**獨立測試**: 可建立具備不同 metadata 的卡牌，進入 `/Cards` 後使用 keyword、meal type 與多個 metadata 條件篩選，確認結果只顯示同時符合所有條件的 active cards，並能保留與清除篩選狀態。

**驗收情境**:

1. **Given** 卡牌庫存在多張不同價格、時間與標籤的卡牌，**When** 使用者選擇「晚餐 + 中價位 + 30 分鐘內 + 清淡」條件，**Then** 系統只顯示同時符合所有條件的 active cards。
2. **Given** 使用者已套用多個篩選條件，**When** 使用者查看結果清單，**Then** 系統必須清楚顯示目前條件，並提供可操作的清除條件入口。
3. **Given** 沒有任何卡牌符合條件，**When** 使用者執行篩選，**Then** 系統顯示目前語系的無結果訊息，且保留可調整條件的操作入口。
4. **Given** 使用者在英文語系下篩選卡牌，**When** 卡牌名稱、餐別與 metadata 呈現於結果清單，**Then** 可見文字必須依目前語系顯示，不得出現未翻譯 key。

---

### 使用者故事 3 - 維護餐點決策資訊 (Priority: P3)

當使用者新增或編輯餐點卡牌時，可以補充價格區間、準備或等待時間、飲食偏好、辣度與自訂標籤。既有卡牌若尚未補齊 metadata，仍必須可被瀏覽、搜尋、編輯與抽卡；使用者可逐步補充資訊，而不被迫一次修完所有卡牌。

**優先級理由**: 條件篩選需要可維護的資料來源；但既有卡牌不能因為新增 metadata 而突然失效或阻擋核心抽卡流程。

**獨立測試**: 可使用既有 schema v3 卡牌資料載入系統，確認未填 metadata 的卡牌仍可正常抽卡與管理；再新增或編輯卡牌填入 metadata，確認資料在重新啟動後保留，並可被首頁與卡牌庫篩選使用。

**驗收情境**:

1. **Given** 使用者正在新增餐點卡牌，**When** 使用者填寫必要雙語餐點內容並選擇價格、時間、飲食偏好、辣度與標籤後送出，**Then** 系統儲存該卡牌並讓這些 metadata 可用於後續篩選與抽卡。
2. **Given** 使用者編輯既有卡牌，**When** 使用者更新 metadata 但不改變餐點名稱、描述或餐別，**Then** 系統保留同一不可變卡牌 ID，既有抽卡歷史與統計不得被切分。
3. **Given** 舊卡牌沒有任何 metadata，**When** 使用者未套用 metadata 篩選進行抽卡，**Then** 該卡牌仍可進入原本符合餐別或隨機模式的候選池。
4. **Given** 舊卡牌沒有某項 metadata，**When** 使用者套用該 metadata 條件篩選，**Then** 該卡牌不應被視為符合該條件，除非使用者已明確補上對應資訊。

### 邊界情況

- 當使用者只套用餐別、不套用 metadata 條件時，抽卡與搜尋必須維持既有行為。
- 當使用者套用多個 metadata 條件時，系統必須以「同時符合全部條件」作為篩選規則。
- 當使用者套用多個標籤時，預設必須只顯示同時具有所有所選標籤的卡牌。
- 當篩選後候選池為空時，系統不得抽出替代卡牌，也不得增加成功抽卡歷史或統計。
- 當卡牌為已刪除狀態時，不得出現在卡牌庫一般篩選結果或未來抽卡候選池，但既有歷史統計仍依 004 規則保留。
- 當使用者語系切換時，篩選條件與已揭示結果不得被清除；metadata 顯示名稱必須依目前語系呈現。
- 當使用者修改 metadata 後，既有 drawHistory 必須仍綁定同一 card ID，不得因 metadata 變化建立新歷史身分。
- 條件篩選不得引入權重、偏好加成、保底、稀有度、點數、付費或任何改變候選池內等機率的規則。
- 本功能不包含外部店家 API、地圖、地址驗證、營業時間判斷或自動抓取餐點資訊。

## 需求 *(mandatory)*

### 功能需求

- **FR-001**: 系統必須允許每張 active 餐點卡牌保存可選的決策 metadata，至少包含自訂標籤、價格區間、準備或等待時間區間、飲食偏好與辣度。
- **FR-002**: 系統必須在首頁抽卡表單提供 metadata 篩選控制，使使用者能在正常模式與隨機模式抽卡前縮小候選池。
- **FR-003**: 正常模式套用篩選時，系統必須先依餐別限制候選池，再套用所有 metadata 條件。
- **FR-004**: 隨機模式套用篩選時，系統不得要求餐別，並必須從全部 active cards 中套用所有 metadata 條件。
- **FR-005**: 任一篩選後候選池包含 N 張有效卡牌時，系統必須讓每張候選卡牌維持相同標稱抽中機率 `1/N`。
- **FR-006**: 篩選後候選池為空時，系統不得宣告抽卡成功、不得新增成功抽卡歷史、不得改變統計，並必須顯示可理解的空候選池訊息。
- **FR-007**: 系統必須在抽卡結果中顯示被抽中卡牌的相關 metadata 摘要，使使用者能理解該結果為何符合目前條件。
- **FR-008**: `/Cards` 卡牌庫必須支援 keyword、meal type 與 metadata 條件的組合篩選。
- **FR-009**: `/Cards` 篩選結果必須只包含同時符合所有已套用條件的 active cards。
- **FR-010**: `/Cards` 必須顯示目前已套用條件，並提供清除條件入口。
- **FR-011**: 卡牌詳情頁必須顯示該卡牌目前保存的 metadata；沒有填寫的 metadata 必須以不誤導的空狀態呈現。
- **FR-012**: 新增與編輯卡牌表單必須允許使用者維護 metadata，但不得讓 metadata 缺漏阻止既有必要雙語欄位的新增或編輯流程。
- **FR-013**: 舊版卡牌資料升級後，缺少 metadata 的卡牌仍必須可被瀏覽、查看、編輯、刪除與未篩選抽卡使用。
- **FR-014**: 缺少某項 metadata 的卡牌在使用者套用該項 metadata 條件時不得被視為符合條件。
- **FR-015**: 自訂標籤必須在儲存與篩選前正規化，至少去除前後空白並避免同一卡牌內出現大小寫或空白差異造成的重複標籤。
- **FR-016**: metadata 的更新不得改變卡牌不可變 ID、抽卡歷史、卡牌狀態或既有統計歸屬。
- **FR-017**: metadata 篩選條件在語系切換、主題切換與表單驗證失敗後必須盡可能保留，不得清除使用者已選條件。
- **FR-018**: metadata 顯示名稱、篩選控制、空狀態、驗證訊息與結果摘要必須依目前 runtime 語系呈現。
- **FR-019**: metadata 篩選不得新增外部 JSON API；公開介面維持 Razor Pages、表單欄位、query string、HTML 狀態與使用者可見訊息。
- **FR-020**: 本功能不得引入權重、偏好加成、付費、下注、點數、稀有度、保底或任何與等機率候選池相衝突的規則。

### 資料完整性需求

- **DI-001**: 篩選條件只能縮小候選池，不得在候選池內改變卡牌抽中權重。
- **DI-002**: 篩選條件必須只作用於 active cards；deleted cards 不得回到一般列表或未來候選池。
- **DI-003**: metadata 更新必須與卡牌更新一樣完整成功或完整失敗，不得留下部分寫入狀態。
- **DI-004**: schema 升級不得遺失既有雙語餐點內容、卡牌狀態、drawHistory 或統計可重建性。
- **DI-005**: 舊卡牌缺少 metadata 時，不得被視為資料損壞；只有 metadata 值格式無效或不支援時才應阻止保存該次變更。
- **DI-006**: 標籤、價格、時間、飲食偏好與辣度的無效值不得持久化；使用者可見錯誤不得包含內部資料、完整 JSON、堆疊追蹤、秘密值或系統提示。

### 非功能需求

- **NFR-001**: 本規格、後續計畫、研究、資料模型、快速入門與任務文件必須使用繁體中文；產品執行時文字依目前核准語系顯示。
- **NFR-002**: 首頁篩選控制、卡牌庫篩選區、卡牌表單與結果摘要必須在桌面、平板與行動尺寸下可操作，文字、按鈕、標籤與表單不得重疊或水平溢出。
- **NFR-003**: 篩選控制必須可用鍵盤、滑鼠與觸控操作，且目前選取條件不得只依賴顏色傳達。
- **NFR-004**: 篩選與抽卡互動不得讓主要頁面超出既有效能預算；常見卡牌數量下，套用篩選與抽卡後應在使用者可感知的短時間內更新。
- **NFR-005**: 篩選失敗、空候選池、資料封鎖、metadata 驗證失敗與寫入失敗必須可診斷，但日誌與 UI 不得包含秘密值、完整資料檔、系統提示或未清理使用者輸入。
- **NFR-006**: 本功能不得降低既有 Anti-Forgery、HTTPS/HSTS/CSP、Razor HTML encoding、雙語 fallback、reduced motion 與抽卡 idempotency 保護。

### 關鍵實體

- **餐點決策資訊 (MealCardMetadata)**: 代表一張餐點卡牌的可選決策欄位，包含自訂標籤、價格區間、準備或等待時間區間、飲食偏好與辣度；用於搜尋與候選池篩選。
- **價格區間**: 使用者可用來描述餐點大致花費的分類值，例如低價位、中價位、高價位；具體顯示文字依目前語系呈現。
- **時間區間**: 使用者可用來描述餐點取得或準備所需時間的分類值，例如快速、一般、較久；不要求自動計時或外部營業資料。
- **飲食偏好**: 使用者用來描述餐點特性的分類或旗標，例如蔬食、清淡、重口味、可外帶；第一版僅作為手動維護與篩選資訊。
- **辣度**: 使用者用來描述餐點辣度的分類值，例如不辣、小辣、中辣、重辣；篩選時依分類值匹配。
- **篩選條件**: 首頁與卡牌庫用來縮小卡牌集合的條件組合，可包含既有 keyword、meal type 與新增 metadata 條件。
- **篩選後候選池**: 某次抽卡在套用模式、餐別與 metadata 條件後可被等機率選取的 active cards 集合。

## 成功標準 *(mandatory)*

### 可量測結果

- **SC-001**: 在具有完整 metadata 的測試資料下，100% 的正常模式篩選抽卡結果都符合所選餐別與全部 metadata 條件。
- **SC-002**: 在具有完整 metadata 的測試資料下，100% 的隨機模式篩選抽卡結果都不要求餐別，且符合全部 metadata 條件。
- **SC-003**: 候選池包含 N 張符合條件的有效卡牌時，100% 的公平性驗收都能證明每張候選卡牌標稱機率為 `1/N`，且沒有依 metadata、歷史統計、語系或顯示順序加權。
- **SC-004**: 100% 的空候選池、資料封鎖、驗證失敗與寫入失敗情境都不新增成功抽卡歷史，也不改變統計數字。
- **SC-005**: 90% 的使用者能在首頁 30 秒內套用至少一個條件並完成一次有效抽卡或看到可理解的空候選池提示。
- **SC-006**: 90% 的使用者能在卡牌庫 30 秒內使用 metadata 篩選找到符合條件的卡牌或看到可理解的無結果提示。
- **SC-007**: 舊版卡牌資料升級後，100% 既有 active cards 在未套用 metadata 篩選時仍可被原本的正常模式或隨機模式抽出。
- **SC-008**: metadata 更新後，100% 既有抽卡歷史仍歸屬同一 card ID，總成功抽卡次數、單卡抽中次數與歷史機率不因 metadata 變更而切分。
- **SC-009**: 主要頁面在 390x844、768x1024 與 1366x768 檢查尺寸下，篩選控制、卡牌列表、結果摘要與表單 100% 不出現文字、按鈕、標籤或欄位重疊與水平溢出。
- **SC-010**: `zh-TW` 與 `en-US` runtime UI 下，100% 的 metadata 篩選控制、結果摘要、空狀態與驗證訊息都不顯示未翻譯 key。

## 假設

- 本功能是 004 抽卡模式與統計之後的增量；正常模式、隨機模式、成功歷史、統計表、刪除後歷史保留與語系切換規則都已存在。
- 本功能第一版只支援手動維護 metadata，不支援外部店家資料、地圖、營業時間、自動價格偵測或自動分類。
- 未套用 metadata 篩選時，既有抽卡行為不應改變；套用 metadata 篩選時，只縮小候選池。
- 多個篩選條件採交集規則；多個標籤也採同時符合全部標籤規則。
- metadata 欄位是輔助決策資訊，不參與 duplicate detection；duplicate detection 仍依既有雙語名稱、描述與餐別規則。
- 價格區間、時間區間、飲食偏好與辣度的具體選項可在正式 spec/plan 階段固定，但不得改變本草稿定義的核心行為。
- 若未來需要「偏好排序」或「推薦加權」，必須另開規格，且不得混入本功能的等機率篩選抽卡。
