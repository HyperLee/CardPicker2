# 功能規格: 餐點收藏與手動排除抽卡

**建議功能分支**: `007-card-preference-controls`  
**建立日期**: 2026-05-30  
**狀態**: 草稿  
**輸入**: 使用者描述:「深度閱讀 001-006 歷史規格與目前專案後，新增可擴充功能規格書；方向選擇收藏與排除，讓使用者能長期標記喜歡的餐點，並手動排除不想再被抽出的餐點。」

## 釐清事項

### 2026-05-30 會談

- Q: 這次可擴充功能規格書的主軸要選哪一類？ -> A: 收藏/排除。
- Q: 收藏/排除第一版要把哪種行為列為核心？ -> A: 每張卡可標記收藏或排除抽卡；排除長期保存，收藏只協助篩選與辨識，不加權。
- Q: 收藏與排除的切換操作第一版要放在哪些頁面？ -> A: 卡牌庫/詳情可管理狀態，抽卡結果可立即收藏或排除剛抽中的卡牌。
- Q: 被標記為排除抽卡的卡牌，在卡牌庫中應如何顯示？ -> A: 仍預設顯示，保持可搜尋、可編輯、可取消排除，並以狀態徽章清楚標示。

## 使用者情境與測試 *(必填)*

### 使用者故事 1 - 手動排除不想再抽到的餐點 (優先級: P1)

使用者遇到暫時不想吃、近期吃膩、店家不方便或不再適合被推薦的餐點時，可以把該餐點標記為「排除抽卡」。被排除的卡牌仍保留在卡牌庫中，能被搜尋、查看、編輯、取消排除或刪除，但不得再進入未來任何正常模式、隨機模式、metadata 篩選或近期輪替防重複後的抽卡候選池。

**優先級理由**: 手動排除直接補足 006 明確未包含的長期控制需求。近期輪替只能避免短期重複，無法處理使用者主觀不想再抽到的餐點；排除抽卡能提高結果可採用率，且不引入加權或不透明推薦。

**獨立測試**: 可準備多張有效卡牌，將其中一張標記為排除抽卡後，分別執行正常模式、隨機模式、metadata 篩選與啟用近期輪替的抽卡；只要被排除卡牌永遠不出現在候選池或結果中，且卡牌庫仍能顯示並取消排除，此故事即可獨立交付價值。

**驗收情境**:

1. **前提** 一張午餐卡牌目前可被抽出，**當** 使用者在卡牌庫或詳情頁將它標記為排除抽卡，**則** 該卡牌仍顯示於卡牌庫並標示已排除，但後續午餐正常模式不得抽出它。
2. **前提** 一張卡牌已被標記為排除抽卡，**當** 使用者執行隨機模式或套用 metadata 條件抽卡，**則** 系統不得將該卡牌放入候選池，也不得將它作為結果或 fallback。
3. **前提** 使用者已排除某張卡牌，**當** 使用者取消排除，**則** 該卡牌在符合餐別、metadata 與近期輪替規則時可重新進入抽卡候選池。
4. **前提** 使用者排除所有符合目前條件的卡牌，**當** 使用者啟動抽卡，**則** 系統不得產生結果或成功歷史，並提示可取消排除、調整條件或新增卡牌。

---

### 使用者故事 2 - 收藏喜歡的餐點並快速辨識 (優先級: P2)

使用者想保留常吃、喜歡或值得再訪的餐點時，可以把卡牌標記為收藏。收藏狀態必須在卡牌庫、卡牌詳情與抽卡結果中清楚呈現，並可作為卡牌庫篩選條件，協助使用者快速找回偏好的餐點。收藏不得改變任何抽卡候選池內的等機率規則。

**優先級理由**: 收藏是低風險、高可用性的整理能力，能讓使用者建立個人餐點偏好，但若讓收藏影響抽卡機率會破壞歷史規格反覆強調的公平性，因此第一版只作為辨識與篩選。

**獨立測試**: 可將多張卡牌標記為收藏，在卡牌庫套用「收藏」篩選並查看詳情；再進行抽卡公平性驗收，確認收藏卡牌不因收藏狀態取得更高機率。

**驗收情境**:

1. **前提** 卡牌庫中有多張有效卡牌，**當** 使用者收藏其中一張卡牌，**則** 卡牌庫與詳情頁必須以目前語系顯示收藏狀態。
2. **前提** 使用者已收藏多張卡牌，**當** 使用者在卡牌庫選擇收藏篩選，**則** 系統只顯示目前被收藏且符合其他搜尋條件的卡牌。
3. **前提** 一張收藏卡牌與一張未收藏卡牌同時存在於同一候選池，**當** 使用者執行抽卡，**則** 兩張卡牌在候選池內的標稱抽中機率必須相同。
4. **前提** 使用者取消收藏一張卡牌，**當** 使用者重新查看卡牌庫或詳情頁，**則** 該卡牌不再顯示收藏狀態，且資料在重新啟動後保持一致。

---

### 使用者故事 3 - 從抽卡結果立即整理偏好 (優先級: P3)

使用者抽出餐點後，應能直接在結果區收藏剛抽中的卡牌，或將剛抽中的卡牌標記為排除抽卡。這讓使用者在決策當下整理偏好，不必再切到卡牌庫搜尋同一張卡牌。狀態變更必須保留既有抽卡結果與統計，不得因收藏或排除重新抽選、重寫歷史或改變已揭示結果。

**優先級理由**: 偏好整理最常發生在使用者看到結果的當下。結果區操作能降低步驟，但它依賴 P1/P2 的資料規則，因此排在第三優先級。

**獨立測試**: 可完成一次成功抽卡，在結果區收藏或排除該卡，確認目前結果仍顯示同一張卡牌、成功歷史與統計不變，後續抽卡則依最新排除狀態建立候選池。

**驗收情境**:

1. **前提** 使用者成功抽出一張卡牌，**當** 使用者在結果區按下收藏，**則** 系統更新該卡牌收藏狀態並保留目前揭示結果。
2. **前提** 使用者成功抽出一張卡牌，**當** 使用者在結果區將該卡牌標記為排除抽卡，**則** 系統更新該卡牌排除狀態，且該次成功抽卡歷史、統計與輪替快照不得被修改。
3. **前提** 使用者對同一張結果卡快速重複提交收藏或排除操作，**當** 系統處理提交，**則** 最終狀態必須一致，不得產生重複歷史、資料損壞或互相矛盾的狀態。
4. **前提** 使用者切換語系、主題或 reduced motion 設定，**當** 使用者查看已揭示結果與偏好按鈕，**則** 可見文案依目前語系呈現，但結果身分、歷史紀錄、統計與偏好狀態不得被重新計算。

### 邊界情況

- 使用者尚未設定任何收藏或排除時，所有卡牌行為必須與 006 完成後的既有抽卡、搜尋、統計與輪替規則一致。
- 被標記為排除抽卡的卡牌不得進入未來候選池，但仍必須保留為可瀏覽、可搜尋、可查看詳情、可編輯、可取消排除與可刪除的卡牌。
- 排除抽卡不得等同刪除；排除卡牌若已有成功抽卡歷史，統計仍依既有歷史呈現，不得從歷史中移除。
- 被排除卡牌若同時也是收藏卡牌，兩個狀態都必須可見；收藏不得覆蓋排除，排除仍必須阻止未來抽卡。
- 使用者取消排除後，卡牌不得立即保證被抽中，只能在符合目前模式、餐別、metadata 與近期輪替規則時回到等機率候選池。
- 使用者套用卡牌庫偏好篩選與既有關鍵字、餐別、metadata 條件時，系統必須以同時符合全部條件作為結果集合。
- 卡牌庫預設仍顯示排除卡牌，避免使用者找不到已排除卡牌而無法取消排除。
- 所有符合目前抽卡條件的卡牌都被排除時，系統不得自動忽略排除狀態，不得抽出排除卡牌，也不得新增成功抽卡歷史。
- 使用者刪除已收藏或已排除卡牌時，刪除後歷史保留與統計規則必須沿用 004/006；偏好狀態不得讓已刪除卡牌回到一般候選池。
- 語系切換、主題切換、動畫、reduced motion 與表單驗證失敗不得清除尚未送出的偏好變更，且不得改變已持久化偏好。
- 本功能不得新增收藏加權、偏好分數、保底、稀有度、付費加成、推薦模型、外部店家資料或自動學習規則。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系統必須允許每張未刪除餐點卡牌保存收藏狀態。
- **FR-002**: 系統必須允許每張未刪除餐點卡牌保存排除抽卡狀態。
- **FR-003**: 新增或由舊資料升級而來的卡牌，若沒有既有偏好資料，必須預設為未收藏且未排除抽卡。
- **FR-004**: 使用者必須能在卡牌庫與卡牌詳情頁切換收藏狀態。
- **FR-005**: 使用者必須能在卡牌庫與卡牌詳情頁切換排除抽卡狀態。
- **FR-006**: 使用者成功抽出卡牌後，結果區必須提供收藏或取消收藏該卡牌的操作。
- **FR-007**: 使用者成功抽出卡牌後，結果區必須提供排除或取消排除該卡牌的操作。
- **FR-008**: 收藏與排除抽卡狀態變更必須在重新載入頁面與應用程式重新啟動後保留。
- **FR-009**: 被排除抽卡的卡牌不得出現在任何未來抽卡候選池中，包含正常模式、隨機模式、metadata 篩選後候選池與近期輪替防重複處理前後的候選池。
- **FR-010**: 排除抽卡只能縮小候選池；剩餘候選池包含 N 張卡牌時，每張卡牌標稱抽中機率仍必須為 `1/N`。
- **FR-011**: 收藏狀態不得影響抽卡候選池、候選池排序、抽中權重、近期輪替排除集合、統計分母或歷史機率。
- **FR-012**: 卡牌庫必須預設顯示已排除抽卡的未刪除卡牌，並以目前語系清楚標示排除狀態。
- **FR-013**: 卡牌庫必須支援依收藏狀態篩選卡牌。
- **FR-014**: 卡牌庫必須支援依排除抽卡狀態篩選卡牌，至少能辨識全部、可抽、已排除三種集合。
- **FR-015**: 卡牌庫偏好篩選必須可與既有關鍵字、餐別與 metadata 條件組合，且結果只包含同時符合所有條件的卡牌。
- **FR-016**: 卡牌詳情頁必須顯示目前收藏與排除抽卡狀態，並提供可操作的切換入口。
- **FR-017**: 抽卡結果區的偏好操作不得重新抽卡、不得改變已揭示 card ID、不得新增成功抽卡歷史，也不得修改既有成功歷史或輪替快照。
- **FR-018**: 當排除抽卡造成候選池為空時，系統不得宣告抽卡成功、不得新增成功抽卡歷史、不得改變統計，並必須顯示可理解的修正提示。
- **FR-019**: 排除抽卡造成空候選池時，使用者可見提示必須至少提供取消部分排除、調整篩選條件或新增卡牌等下一步。
- **FR-020**: 收藏與排除狀態的切換操作必須以完整成功或完整失敗方式完成，失敗時不得留下部分更新狀態。
- **FR-021**: 使用者對不存在、已刪除或資料封鎖狀態下的卡牌提交偏好變更時，系統必須拒絕操作並顯示目前語系的可理解訊息。
- **FR-022**: 所有收藏、排除、取消收藏與取消排除操作必須使用狀態變更表單與 Anti-Forgery 保護。
- **FR-023**: 收藏與排除相關文案、按鈕、狀態徽章、驗證訊息、成功訊息與空狀態必須依目前核准的 runtime 語系呈現。
- **FR-024**: 語系切換、主題切換、reduced motion 與動畫時間不得改變偏好狀態、抽卡候選池資料事實、成功歷史、輪替快照或統計口徑。
- **FR-025**: 本功能不得新增面向外部系統的資料介面；公開互動維持既有頁面、表單欄位、query string、HTML 狀態與使用者可見訊息。
- **FR-026**: 本功能不得新增收藏專屬抽卡模式、收藏加權、偏好加成、付費、下注、點數、稀有度、保底、推薦分數或自動學習模型。

### 資料完整性需求

- **DI-001**: 收藏與排除抽卡狀態必須綁定不可變 card ID，不得依餐點名稱、描述、翻譯、metadata 或顯示語系判斷。
- **DI-002**: 偏好狀態更新不得改變卡牌不可變 ID、餐別、雙語內容、metadata、刪除狀態、成功抽卡歷史或統計歸屬。
- **DI-003**: 排除抽卡狀態必須在候選池建立時先於 006 近期輪替防重複生效；已排除卡牌不得出現在輪替前候選池中，也不得因輪替規則回到候選池。
- **DI-004**: 收藏狀態不得參與重複卡牌判斷；重複判斷仍依既有雙語名稱、描述與餐別規則。
- **DI-005**: 已刪除卡牌不得回到一般卡牌庫篩選結果或未來候選池；若已刪除卡牌曾有偏好狀態，該狀態不得讓它重新可抽。
- **DI-006**: 被排除抽卡的卡牌若已有成功抽卡歷史，該歷史仍必須參與總成功抽卡次數、單卡抽中次數與歷史機率計算。
- **DI-007**: 排除抽卡造成的空候選池屬於未成功抽卡，不得留下會影響總次數、單卡抽中次數、歷史機率、近期 N 次排除或輪替快照的紀錄。
- **DI-008**: 同一偏好操作重複提交時，系統必須達成一致最終狀態，不得造成互相矛盾的收藏或排除值。
- **DI-009**: 舊資料缺少偏好欄位時不得被視為損壞；系統必須以安全預設值載入，並在後續成功寫入時保存可辨識的偏好資料。
- **DI-010**: 偏好狀態寫入失敗時不得覆蓋原本有效資料；使用者可見錯誤不得包含完整 JSON、堆疊追蹤、秘密值、系統提示或未清理使用者輸入。

### 非功能需求

- **NFR-001**: 本規格、後續計畫、研究、資料模型、快速入門與任務文件必須使用繁體中文；產品執行時文字依目前核准語系顯示。
- **NFR-002**: 收藏與排除控制、狀態徽章、卡牌庫偏好篩選、抽卡結果偏好操作與空候選池提示必須在桌面、平板與行動尺寸下可操作，文字、按鈕、徽章與表單不得重疊或水平溢出。
- **NFR-003**: 收藏與排除控制必須可用鍵盤、滑鼠與觸控操作，且目前狀態不得只依賴顏色傳達。
- **NFR-004**: 偏好狀態顯示與候選池過濾不得讓主要頁面超出既有效能預算；以至少 150 張有效卡牌與 1,000 筆成功抽卡歷史的驗收資料驗證時，首頁 GET、偏好過濾抽卡 POST、卡牌庫偏好篩選與偏好狀態變更的 page handler p95 必須低於 200ms，使用者操作後主要內容應在 1 秒內更新。
- **NFR-005**: 收藏/排除切換、空候選池、資料封鎖、無效卡牌 ID、寫入失敗與重複提交情境必須可診斷，但日誌與 UI 不得包含秘密值、完整資料檔、系統提示、堆疊追蹤或未清理使用者輸入。
- **NFR-006**: 本功能不得降低既有 Anti-Forgery、HTTPS/HSTS/CSP、Razor HTML encoding、雙語 fallback、reduced motion、抽卡 idempotency、資料封鎖、同目錄暫存檔原子寫入與刪除後歷史保留保護。

### 關鍵實體 *(本功能涉及資料)*

- **卡牌偏好狀態**: 代表一張未刪除餐點卡牌的使用者整理狀態，至少包含是否收藏與是否排除抽卡；綁定不可變 card ID 並跨重新啟動保留。
- **收藏狀態**: 使用者用來標記喜歡、常吃或值得再訪餐點的狀態；可用於卡牌庫辨識與篩選，不改變抽卡機率。
- **排除抽卡狀態**: 使用者用來阻止某張未刪除卡牌進入未來候選池的狀態；可取消，且不等於刪除。
- **偏好篩選條件**: 卡牌庫用來縮小卡牌列表的條件，可包含收藏狀態、可抽狀態或已排除狀態，並可與既有關鍵字、餐別與 metadata 條件組合。
- **可抽卡牌集合**: 未刪除且未被標記為排除抽卡的卡牌集合；正常模式、隨機模式、metadata 篩選與近期輪替防重複只能從此集合繼續縮小候選池。

## 假設

- 本功能是 006 餐點輪替防重複抽卡之後的增量；正常模式、隨機模式、成功抽卡歷史、統計表、已刪除卡牌歷史保留、metadata 篩選、近期 N 次輪替、雙語語系與主題切換都已存在。
- 收藏與排除狀態是單機單人本機使用者的長期偏好，保存於同一份本機卡牌資料中；不支援帳號、多人、跨裝置或跨瀏覽器同步。
- 收藏第一版只用於辨識與卡牌庫篩選，不提供「只抽收藏」模式，也不讓收藏卡牌提高抽中機率。
- 排除抽卡是可逆偏好，不是刪除；使用者可取消排除讓卡牌回到可抽集合。
- 卡牌庫預設顯示排除卡牌，因為找得到才容易取消排除；使用者若只想看可抽卡牌，可使用偏好篩選條件。
- 偏好狀態不參與重複卡牌判斷，不改變卡牌名稱、描述、餐別、metadata 或抽卡歷史。
- 偏好狀態在候選池建立時先排除不可抽卡牌，再套用既有模式、餐別、metadata 與近期輪替規則。
- 若未來要支援收藏專屬抽卡、推薦排序、權重、黑名單分類、餐點計畫或自動學習偏好，必須另開規格，並重新定義它們與公平性、統計與資料完整性的關係。
