# 功能規格: 餐點輪替防重複抽卡

**建議功能分支**: `006-card-rotation-cooldown`
**建立日期**: 2026-05-19
**狀態**: 草稿
**輸入**: 使用者描述: 「著重商業邏輯，新增一份開發規格書；方向選擇輪替防重複，讓餐點抽卡避免近期一直抽到相同餐點。」

## 釐清事項

### 2026-05-19 會談

- Q: 下一個商業邏輯擴充方向要鎖定哪個範圍？ -> A: 輪替防重複。
- Q: 近期防重複第一版要用哪種冷卻規則？ -> A: 依最近 N 次成功抽卡歷史排除近期抽中的卡牌。
- Q: 套用防重複後候選池變空時，系統應如何處理？ -> A: 不自動放寬規則；提示使用者降低 N、關閉防重複或調整其他條件。
- Q: 最近 N 次的 N 要如何設定？ -> A: 首頁預設為 3，且本次抽卡可調整。
- Q: 第一版是否包含手動排除或黑名單？ -> A: 先不包含；本版聚焦近期成功抽卡防重複。
- Q: 「避免最近重複」是否預設啟用？ -> A: 預設啟用，但使用者可關閉或將 N 設為 0 以回到既有候選池規則。

## 使用者情境與測試 *(必填)*

### 使用者故事 1 - 避免近期重複並維持公平抽卡 (優先級: P1)

使用者連續使用餐點抽卡決定要吃什麼時，不希望短時間內一直抽到最近已經被推薦過的餐點。系統必須在既有正常模式、隨機模式與決策資訊篩選建立候選池後，再依使用者設定排除最近 N 次成功抽卡中出現的卡牌，最後只在剩餘候選池內維持等機率抽卡。這讓抽卡結果更有輪替感，但不引入加權、偏好分數或任何不透明推薦規則。

**優先級理由**: 這是本功能的核心價值。既有抽卡與 metadata 篩選已能建立符合條件的公平候選池，但仍可能連續抽到近期結果；輪替防重複能提升日常可用性，同時保留「候選池內等機率」的可信規則。

**獨立測試**: 可準備多張有效卡牌與已知成功抽卡歷史，設定最近 N 次排除後執行正常模式與隨機模式抽卡；只要近期抽中的卡牌不進入本次候選池，且剩餘每張候選卡牌標稱機率為 `1/M`，此故事即可獨立交付價值。

**驗收情境**:

1. **前提** 使用者選擇正常模式、午餐，且午餐候選池中有 5 張有效卡牌，其中 2 張出現在最近 3 次成功抽卡歷史，**當** 使用者以預設「避免最近重複」啟動抽卡，**則** 系統只會從剩餘 3 張午餐卡牌中等機率抽出一張。
2. **前提** 使用者選擇隨機模式並套用低價位與不辣條件，且符合條件的大池中有部分卡牌出現在最近 N 次成功抽卡歷史，**當** 使用者啟動抽卡，**則** 系統不要求餐別，並先套用既有篩選條件，再排除近期卡牌，最後從剩餘卡牌中等機率抽出一張。
3. **前提** 使用者關閉「避免最近重複」或將 N 設為 0，**當** 使用者完成抽卡，**則** 系統不得套用近期排除，抽卡候選池必須回到既有正常模式、隨機模式與 metadata 篩選規則。
4. **前提** 同一次成功抽卡操作已經產生結果與歷史紀錄，**當** 使用者重新整理、快速連點或重複提交同一操作，**則** 系統重顯原成功結果，不得因最新 N 值、最新歷史或最新候選池重新抽選。

---

### 使用者故事 2 - 防重複後無可抽餐點時引導放寬條件 (優先級: P2)

使用者套用餐別、metadata 篩選與近期防重複後，可能把候選池縮到沒有任何餐點。系統必須清楚區分「原本就沒有符合條件」與「防重複排除後變成沒有候選」兩種情境；後者不得暗中自動放寬規則或抽出近期重複卡牌，而應提示使用者降低 N、關閉防重複或調整條件。

**優先級理由**: 小型卡牌庫或高限制條件下很容易出現空候選池。若系統自動放寬卻不明說，會破壞使用者對商業規則的信任；若只顯示一般無結果，又會讓使用者不知道下一步怎麼修正。

**獨立測試**: 可準備一個原始候選池非空、但其中所有卡牌都出現在最近 N 次成功抽卡歷史的資料集；確認系統不產生成功結果、不新增歷史、不改變統計，並顯示可操作的放寬提示。

**驗收情境**:

1. **前提** 既有模式、餐別與 metadata 條件可找到 3 張有效卡牌，但 3 張都出現在最近 3 次成功抽卡歷史，**當** 使用者啟動抽卡，**則** 系統不得抽出結果，且必須提示可降低最近次數或關閉避免重複。
2. **前提** 使用者看到防重複造成的空候選池提示，**當** 使用者將 N 從 3 降為 1 後再次啟動抽卡，**則** 系統必須依新 N 值重建候選池，並在有剩餘卡牌時完成公平抽卡。
3. **前提** 既有模式、餐別與 metadata 條件在套用防重複前就沒有任何候選卡牌，**當** 使用者啟動抽卡，**則** 系統必須顯示既有無符合餐點或空卡池訊息，而不是誤導為近期防重複造成。
4. **前提** 使用者送出不支援的 N 值，例如負數、超出上限或非數字，**當** 使用者嘗試抽卡，**則** 系統必須拒絕該次操作、顯示可修正訊息，且不得新增成功歷史或統計。

---

### 使用者故事 3 - 看懂本次輪替規則與歷史一致性 (優先級: P3)

使用者完成抽卡或遇到空候選池時，應能理解本次是否套用近期防重複、排除了幾張近期卡牌，以及最後可抽候選池大小。語系切換、主題切換、reduced motion、卡牌改名、翻譯更新與卡牌刪除都只能改變呈現，不得改變已成立的抽卡結果、歷史紀錄或統計口徑。

**優先級理由**: 輪替規則會改變候選池範圍；若 UI 只顯示結果而不顯示規則摘要，使用者可能誤以為抽卡不公平或某些餐點消失。結果與統計也必須延續既有不可變卡牌身分規則。

**獨立測試**: 可在抽卡成功、空候選池、語系切換、卡牌改名與刪除後重新查看結果與統計，確認顯示文字可變，但抽中卡牌 ID、成功歷史、統計分母與近期排除規則皆保持一致。

**驗收情境**:

1. **前提** 使用者以「避免最近 3 次重複」完成抽卡，且有 2 張近期卡牌被排除，**當** 系統揭示結果，**則** 結果區必須顯示目前套用的近期排除摘要與最後候選池大小。
2. **前提** 使用者切換繁體中文與英文語系，**當** 使用者查看已揭示結果或空候選池提示，**則** 可見文案必須依目前語系呈現，但抽中卡牌、排除規則、歷史紀錄與統計不得改變。
3. **前提** 某張最近抽中的卡牌已被刪除且保留歷史，**當** 使用者啟動新抽卡，**則** 該已刪除卡牌不得進入原始候選池；近期排除規則只會排除仍在本次候選池內的有效卡牌。
4. **前提** 使用者更新卡牌名稱、描述、翻譯或 metadata，**當** 系統計算最近 N 次成功抽卡排除集合，**則** 必須仍依不可變卡牌 ID 判斷近期卡牌，不得因顯示文字變更而失去排除關係。

### 邊界情況

- 使用者尚未有任何成功抽卡歷史時，預設開啟的防重複不得排除任何卡牌，抽卡行為等同既有候選池規則。
- 最近 N 次成功抽卡歷史中同一張卡牌出現多次時，排除集合只包含該卡牌一次；N 仍代表最近 N 筆成功抽卡紀錄，而不是最近 N 張唯一卡牌。
- 最近 N 次成功抽卡歷史包含目前已刪除卡牌時，該卡牌可出現在歷史中，但不得進入新抽卡候選池，也不會造成額外可抽卡牌被排除。
- 最近 N 次成功抽卡歷史包含不同餐別卡牌時，正常模式只會排除同時存在於本次餐別候選池內的卡牌；其他餐別歷史不會額外影響目前餐別候選池。
- 隨機模式不得因畫面保留舊餐別狀態而被餐別限制；近期排除只作用於隨機模式的大池候選卡牌。
- metadata 篩選與近期排除同時存在時，系統必須先套用既有模式、餐別與 metadata 規則，再套用近期排除。
- 防重複後候選池為空時，系統不得自動降低 N、不得自動關閉防重複、不得抽出近期重複卡牌，除非使用者明確修改條件後再次送出。
- 使用者輸入無效 N 值時，系統必須拒絕抽卡，不得把無效值悄悄修正後繼續抽卡。
- 同一次成功操作重複提交時，必須重顯原結果，不得因為該結果本身已成為最新歷史而在 replay 時被防重複排除。
- 防重複設定不得改變既有總成功抽卡次數、單一卡牌抽中次數、歷史機率公式或已刪除卡牌歷史保留規則。
- 華麗賭場或皇室風格文案不得暗示「冷卻後更幸運」、「下一次機率提高」、「保底」或任何未定義機率規則。
- 本功能不包含手動排除、長期黑名單、收藏、評分、餐點排程、偏好加權、推薦分數、外部店家資料或自動學習模型。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 首頁抽卡流程必須提供「避免最近重複」控制，且預設為啟用。
- **FR-002**: 首頁必須提供最近成功抽卡排除次數 N 的控制，預設值為 3。
- **FR-003**: 系統必須允許使用者在本次抽卡中調整 N，且 N 的有效範圍必須固定為 0 到 10。
- **FR-004**: N 為 0 或「避免最近重複」關閉時，系統不得套用近期排除，候選池必須完全依既有正常模式、隨機模式與 metadata 篩選規則建立。
- **FR-005**: N 大於 0 且「避免最近重複」啟用時，系統必須讀取最近 N 筆成功抽卡歷史紀錄，建立近期排除集合。
- **FR-006**: 最近 N 筆成功抽卡歷史必須只包含已宣告成功且已持久化的抽卡紀錄；驗證失敗、空候選池、資料封鎖、未投幣、寫入失敗與重複 replay 不得計入最近 N 次。
- **FR-007**: 近期排除集合必須依不可變卡牌 ID 建立，不得依卡牌名稱、翻譯、描述、餐別顯示文字或 metadata 顯示文字建立。
- **FR-008**: 若最近 N 筆歷史中同一卡牌 ID 出現多次，排除集合仍只保留該卡牌 ID 一次。
- **FR-009**: 系統必須先依既有抽卡模式建立原始候選池：正常模式使用所選餐別的有效卡牌，隨機模式使用全部有效卡牌。
- **FR-010**: 系統必須在原始候選池上套用既有 metadata 篩選條件，再套用近期排除規則。
- **FR-011**: 近期排除規則只能移除同時存在於目前候選池與近期排除集合中的有效卡牌。
- **FR-012**: 若套用近期排除後候選池包含 M 張卡牌，系統必須讓每張候選卡牌維持相同標稱抽中機率 `1/M`。
- **FR-013**: 近期排除不得引入權重、偏好加成、保底、稀有度、分數、連抽加成、付費加成或任何改變候選池內等機率的規則。
- **FR-014**: 若既有模式、餐別與 metadata 條件在近期排除前已沒有候選卡牌，系統必須顯示既有空卡池或無符合條件訊息。
- **FR-015**: 若近期排除前候選池非空，但排除後候選池為空，系統不得產生抽卡結果、不得新增成功抽卡歷史、不得改變統計，並必須顯示防重複造成空候選池的可修正訊息。
- **FR-016**: 防重複造成空候選池時，使用者可見訊息必須至少提供降低 N、關閉避免重複或調整其他篩選條件等下一步。
- **FR-017**: 使用者送出無效 N 值時，系統必須拒絕抽卡並以目前語系顯示可修正訊息。
- **FR-018**: 成功抽卡結果必須顯示本次是否套用避免最近重複、使用的 N 值、近期排除後候選池大小，以及可理解的輪替摘要。
- **FR-019**: 若有卡牌因近期排除被移除，系統必須顯示被排除卡牌數量；是否顯示被排除卡牌名稱可由後續 UI 設計決定，但不得顯示已刪除卡牌為可抽候選。
- **FR-020**: 同一次成功抽卡操作被重複提交時，系統必須重顯原成功結果，不得重新套用目前的近期排除規則或新增歷史紀錄。
- **FR-021**: 卡牌改名、描述更新、翻譯更新或 metadata 更新不得切斷近期排除判斷；近期排除必須仍以原卡牌 ID 生效。
- **FR-022**: 已刪除卡牌不得進入未來抽卡候選池；若已刪除卡牌出現在最近 N 筆歷史中，只能作為歷史事實保留，不得造成不存在於候選池的卡牌被抽出。
- **FR-023**: 語系切換、主題切換、reduced motion 與動畫時間不得改變本次候選池、近期排除集合、抽中卡牌或統計口徑。
- **FR-024**: 「避免最近重複」設定與 N 值不需要跨頁面或跨重新啟動持久保存；本版以首頁本次抽卡表單狀態為準。
- **FR-025**: 本功能不得新增面向外部系統的資料介面；公開介面維持 Razor Pages、表單欄位、query string、HTML 狀態與使用者可見訊息。
- **FR-026**: 本功能不得新增手動排除、長期黑名單、收藏、評分、餐點排程或推薦加權；這些需求必須另開規格。

### 資料完整性需求

- **DI-001**: 防重複規則只能縮小候選池，不得在剩餘候選池內改變任一卡牌抽中權重。
- **DI-002**: 近期排除必須以成功抽卡歷史的不可變卡牌 ID 為唯一判斷依據。
- **DI-003**: 成功抽卡歷史的新增仍必須與抽卡結果一起原子成立；若歷史紀錄無法保存，該次抽卡不得宣告成功。
- **DI-004**: 防重複後空候選池屬於未成功抽卡，不得留下會影響總次數、卡牌抽中次數、歷史機率或最近 N 次判斷的紀錄。
- **DI-005**: 同一 `OperationId` 最多只能產生一筆成功抽卡歷史；重複提交必須讀取已成立結果，不得重新抽選。
- **DI-006**: 最近 N 次排序必須依成功抽卡發生時間由新到舊判斷；若時間相同，持久化順序較後者視為較新。
- **DI-007**: 近期排除集合中的已刪除卡牌、無效卡牌或不存在卡牌不得被抽出；若歷史參照缺失導致資料驗證失敗，必須依既有資料封鎖規則處理。
- **DI-008**: N 的無效值不得持久化為有效設定，也不得被用來建立候選池。
- **DI-009**: 防重複規則不得改變既有統計公式；總成功抽卡次數仍為全部成功歷史紀錄筆數，單卡歷史機率仍為該卡成功抽中次數除以總成功抽卡次數。
- **DI-010**: 任何語系、主題、動畫、顯示排序或結果摘要變更都不得改變已持久化的成功抽卡事實。

### 非功能需求

- **NFR-001**: 本規格、後續計畫、研究、資料模型、快速入門與任務文件必須使用繁體中文；產品執行時文字依目前核准語系顯示。
- **NFR-002**: 首頁防重複控制、N 值輸入、結果摘要與空候選池提示必須在桌面、平板與行動尺寸下可操作，文字、按鈕、數字與狀態標籤不得重疊或水平溢出。
- **NFR-003**: 防重複控制必須可用鍵盤、滑鼠與觸控操作，且啟用狀態不得只依賴顏色傳達。
- **NFR-004**: 近期排除計算不得讓主要抽卡流程超出既有效能預算；以至少 150 張有效卡牌與 1,000 筆成功抽卡歷史的本機 JSON fixture 驗證時，首頁抽卡 handler p95 必須低於 200ms。
- **NFR-005**: 防重複套用、空候選池、無效 N 值、資料封鎖與寫入失敗情境必須可診斷，但日誌與 UI 不得包含秘密值、完整資料檔、系統提示、堆疊追蹤或未清理使用者輸入。
- **NFR-006**: 本功能不得降低既有 Anti-Forgery、Razor HTML encoding、雙語 fallback、reduced motion、抽卡 idempotency、資料封鎖與原子寫入保護。

### 關鍵實體 *(本功能涉及資料)*

- **輪替防重複設定**: 使用者在首頁本次抽卡提交的設定，包含是否啟用避免最近重複，以及最近成功抽卡排除次數 N；本版不跨重新啟動持久保存。
- **最近成功抽卡範圍**: 依成功抽卡歷史由新到舊取出的最近 N 筆紀錄，只包含已宣告成功且已保存的抽卡事實。
- **近期排除集合**: 由最近成功抽卡範圍中的不可變卡牌 ID 去重後形成，用於從本次候選池移除近期卡牌。
- **輪替前候選池**: 依既有正常模式、隨機模式與 metadata 篩選建立的有效卡牌集合。
- **輪替後候選池**: 從輪替前候選池移除近期排除集合後，實際可被等機率抽出的有效卡牌集合。
- **輪替摘要**: 使用者可見的本次規則說明，至少包含是否啟用防重複、N 值、排除卡牌數與輪替後候選池大小。
- **放寬提示**: 當輪替前候選池非空但輪替後候選池為空時，提供使用者可修正下一步的可見訊息。

## 成功標準 *(必填)*

### 可量測結果

- **SC-001**: 在已知成功抽卡歷史資料下，100% 啟用防重複且 N 大於 0 的成功抽卡結果都不屬於最近 N 筆成功抽卡歷史中仍存在於本次候選池的卡牌。
- **SC-002**: 防重複後候選池包含 M 張有效卡牌時，100% 的公平性驗收都能證明每張候選卡牌標稱機率為 `1/M`，且沒有依歷史次數、語系、顯示排序、metadata 或主題加權。
- **SC-003**: N 為 0 或防重複關閉時，100% 抽卡候選池與既有 005 規則一致，不額外排除近期卡牌。
- **SC-004**: 100% 防重複造成空候選池的情境都不新增成功抽卡歷史、不改變統計，並顯示可操作的放寬提示。
- **SC-005**: 100% 原始候選池本來就為空的情境都顯示既有空卡池或無符合條件訊息，不誤判為防重複造成。
- **SC-006**: 100% 無效 N 值提交都被拒絕，不產生抽卡結果、不新增歷史、不改變統計。
- **SC-007**: 同一次成功抽卡操作快速連點、重新整理或重複提交時，100% 重顯同一張原抽中卡牌，且總成功抽卡次數與該卡抽中次數只增加一次。
- **SC-008**: 卡牌改名、描述更新、翻譯更新或 metadata 更新後，100% 近期排除仍依同一卡牌 ID 生效。
- **SC-009**: 已刪除且出現在最近 N 筆歷史的卡牌，100% 不會進入新抽卡候選池；若其不在本次候選池中，也不會降低其他卡牌的可抽資格。
- **SC-010**: `zh-TW` 與 `en-US` 執行時介面下，100% 防重複控制、結果摘要、空候選池提示與驗證訊息都不顯示未翻譯 key。
- **SC-011**: 在 390x844、768x1024 與 1366x768 檢查尺寸下，首頁防重複控制、結果摘要與空候選池提示 100% 不出現文字、按鈕、數字或狀態標籤重疊與水平溢出。
- **SC-012**: 使用至少 150 張有效卡牌與 1,000 筆成功抽卡歷史的驗收資料時，首頁啟用防重複抽卡 handler p95 低於 200ms，且使用者啟動抽卡後主要內容在 1 秒內更新為結果或可理解提示。

## 假設

- 本功能是 005 餐點條件篩選抽卡之後的增量；正常模式、隨機模式、成功抽卡歷史、統計表、已刪除卡牌歷史保留、metadata 篩選、雙語語系與主題切換都已存在。
- 「最近 N 次」指最近 N 筆成功抽卡歷史紀錄，不是最近 N 張唯一卡牌；同一張卡在最近 N 筆中重複出現時，排除集合仍只包含該卡一次。
- N 的第一版有效範圍固定為 0 到 10；0 等同關閉防重複，預設值為 3。
- 防重複設定不作為使用者偏好持久保存；使用者重新開啟網站時，首頁使用預設啟用與 N=3。
- 防重複是候選池縮小規則，不是偏好排序或推薦加權；候選池內仍維持等機率。
- 本功能不提供手動排除、長期黑名單、收藏、評分或餐點計畫；若未來需要，必須另開規格並重新定義其與候選池、公平性、統計的關係。
- 本專案仍維持單機單人本機使用，不處理多人同時操作、帳號同步或跨裝置輪替偏好同步。
